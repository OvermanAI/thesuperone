> 不講虛的。這篇用主角 Smart 從零打造一座**會 BUILD、會 SELL、真的賺錢**的 AI 工廠，
> 一步一步走完：做產品、開模、組生產線、控成本、訂價、每天賣。
> 原則：只寫真實狀態與「示範用」的成本試算，不杜撰收入數字；
> 並誠實標出哪些**已被驗證**、哪些還是**假設**。

這座工廠要量產的產品，叫 **Agent 原生網站**。先把它講清楚，後面才不會是空話。

---

## 一、產品定義：什麼是「Agent 原生網站」

大部分中小企業的官網是一張**死掉的海報**：放著，沒人顧，訪客來了自己摸、摸不到就走。

Smart 要賣的不是這個。他賣的是 **Agent 原生網站**——核心不是頁面，是站內一顆 **24/7 的 AI 業務員**：

| | 傳統官網 | Agent 原生網站 |
|---|---|---|
| 本質 | 靜態海報 | 會接客的業務員 |
| 訪客來了 | 自己看、看不懂就走 | AI 主動接待、回答 |
| 老闆睡覺時 | 網站發呆 | AI 還在篩客、留資料 |
| 賣的東西 | 「一個網站」 | 「多接幾個客」 |

站內這顆 Agent 做三件事：
1. **答客**：用這家店的資料回答（價格、服務、營業時間）
2. **篩客**：主動問需求，分辨出有預算、有意願的人
3. **收口**：要到聯絡方式，或直接約時間

技術組成（具體、可建，不是概念）：
```
Next.js 靜態網站（部署到 Vercel）
  + 聊天 widget → 後端 /api/chat → 客服模型（用客戶資料當 system prompt）
  + 約時間（嵌入式日曆）+ 留資料（寫進表單／試算表）
```

**這就是要量產的產品。** 中小企業老闆聽得懂、付得下去——因為它直接對應「多接幾個客」，不是一筆看不到回報的網站費。

## 二、開模：先給自己做一個母版（dogfood）

實體工廠開發新產品的第一個動作叫**開模**——先做出一個標準母版，後面才能照著它量產。AI 工廠一樣。

但 Smart 的關鍵判斷是：**母版的第一個用戶，是他自己。**

他不先接客戶，而是先把自己的一個站升級成 Agent 原生——例如在自己的學習教練站，裝一顆「課程顧問 Agent」：訪客問「我該從哪開始」，Agent 用他的課程資料回答、判斷對方程度、收口約一次諮詢。

**為什麼一定先 dogfood，不直接賣客戶：**
- 用**自己的錢**踩出真實的每月成本（客服模型一個月到底花多少、快取有沒有用）
- 用**自己的流量**踩出真實的轉換（這顆 Agent 到底有沒有幫他多收幾個人）
- 母版被自己用過、修過，才算定型——這才是日後敢拿去賣的底氣

> **作者實況（已驗證）**：這套「先做出一個能複製的母版，再複製成多站」，作者在自己的 1＋7 個網站上**真的做過 8 次**——先磨出一個站的母版（內容＋發布＋部署一條龍），再複製衍生到其餘站。所以「從一個母版量產多站」這件事，不是賭，是跑著的東西。
>
> 缺的那一塊很誠實：1＋7 是作者**自己的**媒體資產，不是賣給客戶的產品。要變成能賣的工廠，母版要從「內容站」換成「Agent 原生網站」，而且前面要接上「客戶進料」——這一段，是 Smart 這個案例要往前推的假設。

## 三、抽規格：把「料」和「機台」切開

母版之所以是母版，靠的是一件事：**會變的部分，被乾淨地抽出來。**

Smart 把母版拆成「不變的機台」和「每個客戶會變的料」，後者收進一份 `client-spec.json`：

| 不變的機台（寫死在 template） | 每個客戶會變的料（spec 欄位） |
|---|---|
| 四頁結構、聊天 widget、`/api/chat`、部署流程、QA 截圖腳本 | 公司名、服務項目、FAQ、語氣、約時間連結、品牌色 |

```json
// client-spec.json —— 這就是工廠的「進料單」
{
  "businessName": "",
  "services": [],
  "faq": [],
  "tone": "專業 | 親切",
  "calLink": "",
  "brandColor": ""
}
```

**切得越乾淨，生產線越能跑。** 切不乾淨——每個客戶都要動到機台——你就退回去當接案工，不是開工廠。這份 `client-spec.json` 是整座工廠的交接介面：客戶填它、BUILD 線吃它、SELL 線也借它生 demo。

## 四、組生產線：用「梯隊／角色」組織 Agent 工作團隊

這是整篇的核心，也是 Smart 一開始想錯、後來改對的地方。

他原本以為「用一家最強的 AI 全包」就好。錯。**那會同時輸掉效率和利潤**——因為沒有任何一家廠商，在所有任務上又最強又最便宜。真正會賺錢的工廠是一支**異質艦隊**：每個任務，路由到「能過這個任務品質門檻的、最便宜的 Agent」。

組織原則一句話：

> **能寫死的用腳本（0 成本）；需要判斷的，配當下最經濟、剛好夠用的 Agent；最難的，才動用當下最強的雙核；需要品味的，留給你自己。**

Smart 的工廠分成五個梯隊（**用角色描述，不綁特定型號**）：

| 梯隊（角色） | 用什麼等級的 Agent | 負責 |
|---|---|---|
| **最強編程雙核**（premium） | 當下業界並列最強的**兩顆**編程核心 | 編排、最難的建站、品質把關——最硬的 20% |
| **圖片生成**（專科雙核） | 當下最強的兩個圖片生成 Agent | 站內主視覺、產品圖、案例視覺 |
| **影片生成**（專科雙核） | 當下最強的兩個影片生成 Agent | 短影音廣告、產品演示 |
| **經濟主力** | 當下最便宜夠用的模型 + 開源自架 Agent | 解析、分類、量產草稿、站內客服——最量的 60% |
| **免費機台** | 純腳本（0 LLM） | 組裝、部署、寄信、跑分 |

**為什麼最強核心要「雙核」並用？** 不同核心強項互補，而且能**互相交叉驗證**——一顆寫、一顆審，像請了兩個不同學派的資深工程師。它們是 premium、用在最硬的地方；**經濟層才是量產苦工**，兩者不能混為一談。

> **為什麼這篇不寫出模型的名字？**
> 因為型號每隔幾個月就被更新、更強的取代——今天最強的那顆，半年後可能只排第三。
> **會過時的是型號，不會過時的是「混編當下最強 × 最經濟 × 專科」這套組織法。**
> 你要練的不是記住哪個模型最強（那會一直變），而是隨時把手上任務，
> 路由到當下最划算的那顆 Agent。這正是本書講的、唯一會隨 AI 變強而增值的技能。

### BUILD 團隊（生產線）

```
廠長（最強核心）── 編排整條工作流、決定哪個任務派給誰
  ↓
資深工程（第二顆最強核心）── 最難的建站任務 + 交叉驗證
規格解析（經濟層）──────── 客戶雜亂表單 → 乾淨的 client-spec.json
文案（經濟層草稿 → 最強核心潤）─ 依品牌語氣填頁面文案 + SEO
媒體 Agent（圖片／影片專科）── 站內主視覺、產品圖
組裝 + 部署（腳本，0 成本）── 樣板替換、上 Vercel、綁網域
L2 驗收（腳本 + 視覺判斷 + Smart）─ 連結/跑分用腳本；好不好看，Smart 必看
站內客服（經濟層 + prompt caching）─ 交付後 24/7 跑在客戶站上
```

### SELL 團隊（銷售線）

```
業務主管（最強核心）── 管名單→demo→開發信→跟進的排程
  ↓
名單（腳本抓 + 經濟層初篩）──── 找「有店、但官網爛/沒有」的商家
demo 生成（借 BUILD 線）─────── 只生首頁、用經濟層，把投機成本壓到最低
開發信（經濟層草稿 + 最強核心潤前段）─ 點名對方官網的問題 + 附他的 demo 連結
媒體素材（圖片/影片專科）────── 案例視覺、短影音廣告
跟進（腳本寄序列 + 經濟層分類）── 自動序列；有人回 → 分類丟給 Smart
案例內容（經濟層量產 + 潤稿）── 每交付一站 → 自動生案例 + 社群貼文
成交 ──────────────────── Smart 本人（不可外包）
```

## 五、兩線怎麼分工合作

生產線和銷售線不是各跑各的，是**一個會自己轉的循環**。三個交接點：

```
        ┌──── 共用介面：母版（模具）+ client-spec.json 格式 ────┐
        │                                                      │
  SELL 成交 ─→ spec 丟進 BUILD 線 ─→ 產出真站 ─→ 交付           │
        ↑                                          │           │
        │                                          ↓           │
  SELL 拿去當彈藥 ←── 案例 Agent 把交付站變成新素材 ───────────┘
        │
  SELL 又借 BUILD 線 ─→ 批次生 prospect 的 demo ─→ 帶動下一批成交
```

1. **SELL → BUILD**：成交後，SELL 的 `client-spec.json` 直接餵進 BUILD 生產線，同一份格式，無縫交接。
2. **BUILD → SELL**：每個交付的真站，自動被案例 Agent 變成新素材——**產品本身就是行銷**。
3. **共用模具**：SELL 為了「先建再賣」，直接借 BUILD 的生產線去生 demo——**所以兩隊用的是同一台機器**，這正是「BUILD 成本低到可以拿來當 SELL 彈藥」的原因。

## 六、成本與訂價

**Smart 的成本分兩種，先分清楚：**

- **一次性建站成本**（做一次）：用最強核心 + 經濟層建一個站，消耗的 token 約**幾美元**——而且月費已經付了，邊際成本接近零。
- **每月持續成本**（命門）：站內客服 Agent 跑對話的 API 費。用**經濟層模型 + prompt caching**（客戶資料是固定的 system prompt，快取後幾乎不重複計費），一個小企業站一個月壓在**台幣幾十元**等級，再加一點 hosting。

**訂價模型：一次性建站費 + 月費（這才是工廠，不是接案）**

```
建站費（一次性）  ← 覆蓋你的時間 + 建立信任
月費（recurring）← 必須 > 單站每月成本的數倍；累積成 MRR = 工廠真正的價值
```

定價邏輯，抓在兩條線之間：
- **下限＝你的每月成本**（API + hosting），這是地板，月費絕不能低於它
- **上限＝客戶「多接一個客」值多少**：一個裝潢／牙醫／補習班多成交一單可能值幾千上萬，月費對他是划算的
- **賣的是結果（多接客），不是賣 token 成本**——所以價格和你的成本脫鉤，這才有工廠的槓桿

> 示範試算（**示範用，非實際成績**）：若月費定 NT$1,500、單站每月成本約 NT$100 →
> 毛利率 >90%；30 個客戶 = NT$45,000/月的 MRR。
> （真實數字要 Smart 自己跑出來，這裡只示範「成本結構」怎麼算。）

**成本控制的開關**：客服一律用經濟層（不要用最強核心跑客服，貴又沒必要）、開 prompt caching（省最多）、對話設長度上限防爆量、靜態站 + serverless（沒有 24 小時開機的伺服器成本）。

## 七、每天真的賣得出去（誠實版）

先講最重要的誠實話：**沒有「全自動每天自動成交」這種事。** 工廠不是讓你不用賣，是讓你**有能力每天大量出擊**。

**Smart 的殺招：先建好，再賣（Build-before-buy）**

因為建站的邊際成本接近零（只是 token），他可以**在客戶付錢之前，就先幫對方把網站建好**：

```
每天的銷售生產線：
1. 抓名單：找「有店、但官網爛或根本沒有」的中小企業
2. 工廠自動建 demo：把該商家公開資料餵進腳本 → 生一個「他家的」
   Agent 原生網站 demo（網址帶他的店名；只先生首頁，壓成本）
3. 開發信 Agent 寫個人化信：
   「我幫貴店做了一個會自己接客的新官網，這是連結 [demo]。
    要上線嗎？月費 NT$X，我幫你綁好網域。」
4. 寄出 → 對方點進去看到的是「已經做好的、他自己的網站」
```

**這就是為什麼 BUILD 和 SELL 是同一座工廠**：建站便宜到可以拿去當銷售素材本身。Smart 寄的不是報價單，是成品。

每天的實際節奏：
- **工廠出**：N 個 prospect 的 demo + N 封個人化開發信（Agent 草擬，Smart 掃一眼放行）
- **Smart 親手做**：審那幾封信、回覆有興趣的、上成交電話

> **誠實的數字現實**：這是個轉換率遊戲——回覆率低個位數 %、成交更低，早期一定慢。
> 這篇**不假裝有一個漂亮的轉換率**，那個數字要 Smart 自己跑出來。
> 工廠給的是「每天能出 10 個成品提案」的產能，而不是雙手一天只能做 1 個。
> 量夠大，成交才會變成每天的事。

## 八、誠實的代價：異質艦隊的「整合稅」

異質艦隊紙上效率高，但它不是免費的。把它說成真實可行，就得承認代價：

- 你現在要管**多家 API／金鑰／速率限制／輸出格式差異／品質落差**
- 「哪個任務路由到哪顆模型」這套**路由邏輯本身要維護、會漂移**
- 跨模型的輸出品質不一致 → **L2 驗證的負擔變更重**

> **作者實況（未達 Factory Fit 的誠實案例）**：作者光是用一條工作流，
> 讓最強核心去自動化一個 blog 的內容生產，**跑了三週，品質還是差強人意**——
> 卡點正是「設計／文字的主觀品質難以工程化」，也就是缺一道夠強的 L2 驗證迴圈。
> 同一道牆，在多模型的異質艦隊裡只會放大。

所以這座工廠**不是配好就自動賺錢**。它把複雜度從「模型」轉移到「**編排與驗證**」——而這恰好就是本書講的 Boss Engineering 與 Loop Engineering 真正值錢、也真正難的地方。它是你要持續顧的一條產線，不是一個按鈕。

## 九、已驗證 vs 假設（這篇的誠實骨架）

| | 內容 | 狀態 |
|---|---|---|
| **已驗證**（作者親身） | 一個母版 → 複製多站（1＋7 做過 8 次） | ✅ 跑著的 |
| | 本書的「公開寫作」工廠（一條上架+修訂工作流） | ✅ 已達 Factory Fit |
| | 圖片／影片用「最強專科雙核」實際在生產 | ✅ 在用 |
| **未達 / 假設** | blog 內容自動化（主觀品質難工程化） | ⚠️ 三週仍差強人意 |
| | 把母版換成「Agent 原生」、前面接客戶進料 | ⚠️ Smart 案例要往前推的 |
| | 客戶進料多樣性、設計品味 L2 關、每日成交轉換率 | ⚠️ 尚待驗證 |

---

> 一座 OPC 的 AI 工廠長這樣：**BUILD 便宜到能當 SELL 的彈藥（先建再賣）；月費把一次性接案變成可累積的 MRR；五個梯隊混編當下最強 × 最經濟 × 專科，把每塊錢花在刀口上。**
> 你的雙手一天做 1 個站，工廠一天出 10 個提案——這個差距，就是一個人對打一間公司的地方。
> 而代價，是你要學會當那個顧著整條產線的人。輪到你了，Smart。
