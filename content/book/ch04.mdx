> STAGE 2 開啟 — 你已經是老闆了。現在，蓋一座替你工作的工廠
> 警告：這一章要先打掉一張你腦中很爽、但會害你燒光錢的圖 _

CEO 的身分你拿到了。這一關開始動土蓋工廠。

但在搬第一塊磚之前，必須先拆掉一個幻想——一個聽起來很狂、實際上在現在的 AI 能力下**注定失敗又燒錢**的畫面。先把它拆乾淨，你才不會把錢丟進水裡。

---

## CH04-1｜先殺掉一張會害死你的圖

很多人（包括上一版的我）想像中的 AI 工廠長這樣：

```
            你
   ┌────┬────┼────┬────┐
  A1   A2   A3  ...  A1000
（一個你，直接平行指揮 1000 個自主 Agent）
```

這張圖很誘人，因為它讓你以為：任命一個超強的 Claude Code，把一千個 Agent 掛上去，然後它們就自動、免費、不停地把活幹完。

**這張圖是錯的。而且不是小錯，是會讓你燒光預算的根本性錯誤。**

為什麼？下一節講清楚——這牽涉到 Agent 系統裡最硬的物理限制。

---

## CH04-2｜為什麼「一個 Claude Code 直接管一千個」一定爆掉

關鍵限制只有一個詞：**context window（上下文窗口）**。

> 在任何 agent 系統裡，context window 是最稀缺的共享資源。

一個 Agent 要運作，它的窗口裡要塞：工具的說明、每次工具回傳的結果、它自己的推理痕跡、對話歷史。**這些全部都吃 token、全部擠在同一個有限的窗口裡。**

現在想像你讓一個 Claude Code 直接對接一千個 Agent：

```
1000 個 Agent 的工具描述、回傳、狀態回報
        全部灌進「同一個」Claude Code 的窗口
                ↓
   十分鐘的協調對話，窗口就被塞爆
                ↓
   它開始砍掉前面的內容（忘記前文）→ 品質崩
   或瘋狂吞 token 重讀 → 帳單爆炸
```

這不是調個參數、寫個更好的 prompt 就能解決的。**這是結構錯了。** 一個窗口塞不下一千個 Agent 的運作，就像一個人的腦袋同時記不住一千場對話——再聰明也沒用。

你直覺是對的：一個 Claude Code 直接管一千個，注定浪費錢。正解是下一節的結構。

---

## CH04-3｜真實架構：階層 + 隔離 + 外部狀態

會運作的工廠，長這樣——**階層式**，不是平行式：

```
你（Big Boss）         出 10%：願景、方向、高風險審核
   ↓ 只對接一個
AI CEO ＝ Orchestrator（協調者）
   只負責「拆解、啟動、調度、收斂」，不把一千段對話塞自己窗口
   ↓ 分派給
中間管理層：sub-orchestrators / 範圍明確的工作流
   ↓ 各自啟動
短命 sub-agents：每個有「自己獨立的 context window」，做完即丟
```

讓它不爆窗、不燒錢的，是三個機制：

1. **context 隔離**：每個 sub-agent 在自己獨立的窗口裡跑，彼此不互相污染。orchestrator 只收「結果摘要」，不收一千份完整過程。一千個 Agent 的負擔，被切成一千個小窗口分攤，而不是壓在一個窗口上。
2. **外部持久狀態**：工廠的進度、記憶、知識放在「外部的狀態資料庫」，不是塞在對話裡。需要時才撈相關的一小段進窗口。
3. **context 管理**：長流程要靠摘要壓縮（把舊對話壓短）、優先級淘汰（丟掉過時的內容）、分階段載入（詳細指引只在用得到時才進窗口）。

這就是為什麼工廠能變大、你卻不會爆掉——你（和 AI CEO）對工廠的介面，永遠是「收斂過的一對少數」，不是「直連一千」。

## CH04-4｜真相：大半「Agent 工作」其實是寫死的程式碼

還有一個業界老手才懂、新手都忽略的真相：

**「一千個 Agent」裡，真正用到 LLM 判斷的，只佔一小部分。** 大部分工作，其實是**寫死的確定性程式碼**。

```
能用規則 / 腳本 / 既有 API 做的 → 寫死（確定性管線）
   → 更便宜、更快、不會幻覺、結果可預測

只有「真的需要判斷」的節點 → 才呼叫 Agent
```

舉例：一條內容產線裡，「抓資料、排版、發布、寄信、記錄數據」這些都能寫死；只有「寫出有觀點的草稿」「判斷這篇值不值得發」才需要 LLM。

> 核心心法：**LLM 是更大軟體系統裡的一個元件，不是獨立應用。**

所以「擁有一千個 Agent 的工廠」的真實樣貌是：少數 orchestrator + 大量確定性管線 + 在判斷點才啟動的短命 agent。把能寫死的都寫死，是省 token、提可靠度的第一原則。

---

## CH04-5｜這要花錢、要工程：成本與難度的真話

我必須對你誠實，因為這正是其他 AI 創業書不敢講的：

**這座工廠不是免費的，也不是一鍵的。**

- **token 會燒**：fan-out（分派越多 sub-agent）× 長 context × 重試，token 消耗很快。成本控制本身就是核心技能：context 隔離（省窗口）、model routing（便宜的模型做便宜的事，別什麼都用最貴的）、能確定性化就確定性化。
- **這是一門持續工程**：Harness（工具、護欄、環境）要設計、要維護；你的判斷要透過**創辦人知識庫**餵給 AI CEO，它才會用「你的方式」調度。這些都要時間。
- **連管好「一個」Agent 都很難**：以我自己的實測，用 Claude Code 搭 harness 去管「一個」模型做 blog 自動化，到第三週品質都還只是差強人意。所以「管一千個」從來不是「數量問題」，是「工程紀律問題」——架構對了、機制到位，數量才有意義；架構錯了，一個都管不好。

這不是要嚇退你。相對於請一百個人，這仍然是作弊碼——只是這個作弊碼要用一門新工程學去換，不是許願就有。

## CH04-6｜今天的第一步：畫出你的指揮鏈，標出「哪些根本不用 Agent」

別急著任命一個 Claude Code 就走人。今天做一件更值錢的事：

> 挑你工廠的**一條產線**，把它要做的每個步驟列出來，然後為每一步標記：
> - 🟩 **可以寫死**（規則／腳本／API 能解決 → 確定性，不花 LLM token）
> - 🟨 **需要 Agent 判斷**（真的要 LLM 才能做）
> - 🟥 **需要人工檢查點**（高風險，你要親自看）

你會驚訝地發現：一大半其實是 🟩。

這張圖就是你的工廠藍圖——它同時告訴你「該把哪個最強 Agent 放在哪個判斷節點」，以及「哪裡根本不該燒 token」。這是省錢、也是讓工廠真的跑得動的第一步。

---

你現在有了真實的工廠結構：階層調度、context 隔離、外部狀態、能寫死就寫死、在判斷點才用 Agent。

但結構會跑，不代表會「越跑越好」。要讓它自我改善，靠的不是許願，是一門具體的工程——**Loop Engineering。**
