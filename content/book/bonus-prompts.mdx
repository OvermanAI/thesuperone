> 這本書用過、驗證過的提示詞。複製、替換你的主題，就能開始寫你自己的書 _

搭配[贈品一的 WIP 寫作法](/book/bonus-wip)使用。下面 10 條提示詞，覆蓋從「搭站」到「上架」的整個電子書創作生命週期。每一條都可直接貼進 Claude Code。

---

## 0｜從零搭一個 WIP 電子書網站

> **用途**：一句話請 Claude Code 把整個發布平台搭起來。

```md
你是我的全端工程師。幫我從零搭一個「公開寫作」電子書網站：
- 技術：Next.js（App Router）+ MDX + Tailwind，靜態、無資料庫
- 頁面：首頁（書名+簡介+CTA）／目錄（依章節 status 顯示徽章）／
        章節頁（讀 content/book/<slug>.mdx）／建造日誌
- 內容模型：每章 frontmatter 有 title/stage/chapter/slug/status/updated/
            summary/revisions（修訂紀錄清單）
- 章節頁要顯示「修訂紀錄」（最新在上）
- 設計：極簡、克制、好讀（中文長文 66ch 欄寬）
先給我專案結構與你要建的檔案清單，我確認後再開始。
```

## 1｜定義書的定位與受眾

> **用途**：寫之前先把座標釘死，避免寫一本沒人要的書。

```md
你是我的出版策略顧問。我要寫一本關於【你的主題】的電子書。
請用提問引導我（一次一題），釐清並輸出一份「定位文件」：
1. 這本書回答讀者哪一個最痛的問題？
2. 目標受眾是誰（具體到身分、處境、現在卡在哪）？
3. 讀者讀完會「從 ___ 變成 ___」？
4. 和市面上同類書最不一樣的一個主張是什麼？
5. 為什麼是「現在」非讀不可？
問完後幫我整理成一頁定位文件，並指出最大的風險或盲點。
```

---

## 2｜生成目錄結構

> **用途**：把定位變成一條讀者走得通的學習路徑。

```md
依我的定位文件，幫我設計這本書的目錄結構：
- 用「讀者的轉變」當主線，把全書切成 3-4 個 STAGE，每個 STAGE 3-4 章
- 每章給：標題、一句話摘要、讀完帶走什麼
- 確保章與章之間是「前一章的結論，是後一章的前提」
- 標出哪幾章是核心、哪幾章是支撐
先給我骨架，我們對齊後再細修。
```

---

## 3｜把「好」變成可驗證的寫作標準

> **用途**：給 AI 一個能自我檢查的靶——這是內容品質的命門。

```md
我們要替這本書定一份「合格章節」的可驗證標準（rubric），
之後每章寫完都用它自檢。請幫我列出 5-7 條「一章要算合格，必須做到」
的硬條件，要具體、可逐條判斷（不要『要寫得好』這種廢話）。例如：
- 開頭三段內就點出讀者的具體痛點
- 每個主張都附「怎麼做 + 真實成本/限制」，不只宣告
- 結尾給一個讀者今天能做的具體動作
也列出「不可出現」清單（空泛形容詞、AI 腔套話、未驗證的數字…）。
```

---

## 4｜寫一章（帶語調與標準）

> **用途**：產出符合你語氣、且通過 rubric 的章節草稿。

```md
幫我寫【第 N 章：章名】的草稿。
- 語調：【描述你的語氣，例：宣告式、口語、給專業讀者】
- 受眾：【你的讀者】
- 這章要讓讀者得到：【帶走的東西】、改變：【從___到___】
- 寫完後，拿我們的 rubric（見上）逐條自檢，沒過的地方標出來並修
- 用編號小節，每節要有料（機制 → 例子 → 可執行步驟），不要只列大綱
寫好先給我前 1/3，我確認語氣對了，你再寫完。
```

---

## 5｜論證而非宣告（嚴謹度檢查）

> **用途**：把「相信我」的句子，改成「讓讀者自己點頭」。

```md
請審這一章，找出所有「直接宣告、沒有論證」的主張
（例如『這樣做你會成功』『這很簡單』）。對每一處：
1. 指出讀者會在這裡產生什麼懷疑
2. 改寫成：帶可行性 + 真實成本/限制 + 正面回答那個懷疑
3. 凡是收入/成績數字，若我沒提供真實數據，一律拿掉或改成結構性說法
目標：讀者每讀一句是「被說服」，不是「被叫去相信」。
```

---

## 6｜全書術語與骨架一致性校準

> **用途**：多章寫下來難免漂移，用這個對齊整本書。

```md
這是我目前全部章節。請做一次一致性校準：
1. 列出全書關鍵術語，標出哪些章用法不一致、或前後定義衝突
2. 檢查每章是否都扣回全書主線（前一章結論=後一章前提）
3. 找出重複論述、和該合併的段落
4. 給我一份「校準清單」：哪章該改什麼，按優先序排
先別動手改，先給清單讓我決定。
```

---

## 7｜讀後復盤：每章價值審核

> **用途**：用讀者的眼睛，逐章檢驗「值不值得讀」。

```md
模擬我的目標讀者做「讀後復盤」。假設這本書售價 X 元，
請逐章誠實判定：讀完這章「得到什麼、改變什麼」，
以及「這一章是否獨立值回 X 元」（達標／勉強／不足）。
不足的章，指出缺什麼、該補一個什麼「帶得走的工具」
（模板／清單／提示詞／可驗證標準）。不要客氣，照實說。
```

---

## 8｜修訂某一段（保留語氣）

> **用途**：精準改一段，不破壞周圍語氣。

```md
只改這一段：【貼上原文】
問題是：【太抽象／沒例子／太長／語氣不對…】
請在「不改動周圍語氣」的前提下重寫，並用一句話說明你改了什麼、為什麼。
給我 2 個版本讓我挑。
```

---

## 9｜上架並留下修訂紀錄（WIP 核心）

> **用途**：發布不覆寫歷史，每次都留下公開的「改了什麼」。

```md
把【章節 slug】上架到網站：
1. 從 Obsidian 讀這章內容，清掉 Obsidian 專屬語法、移除正文頂部重複標題
2. 補/沿用 frontmatter；status 設【drafting/published】
3. 「保留」既有的 revisions，在最後「追加」一筆：
   - date：今天
   - note：這次改了什麼（一句話）
4. 寫入網站、git commit + push
做完告訴我這章現在第幾筆修訂紀錄。
```

## 怎麼用這份贈品

```
搭站（提示詞 0）
  → 定位 + 目錄（1、2）
  → 定標準（3）
  → 一章一章寫（4）+ 嚴謹度檢查（5）
  → 多章後校準（6）+ 價值審核（7）
  → 持續修訂（8）+ 上架留紀錄（9）
```

把【你的主題】填進去，你就有了一條和我寫《The Super One》一模一樣的生產線。
